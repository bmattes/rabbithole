How should we build the web version?

Three approaches — trade-offs between speed and code reuse

A

Expo Web (React Native Web)

Add web to the existing Expo app's platforms. Most components compile to the browser automatically. PuzzleCanvas needs a mouse/touch shim replacing PanResponder. Haptics becomes a no-op. Deploy via npx expo export --platform web to Cloudflare Pages.

Pros

  • Shares all game logic + hooks
  • One codebase
  • Puzzle data layer unchanged

Cons

  • PuzzleCanvas needs web event rewrite
  • RN Web has occasional quirks
  • Harder to customise look for web
B

Standalone React web app

New web/ directory: plain React + Vite. Port just the puzzle mechanic — no RN dependencies. Canvas rendered with HTML/CSS divs + SVG lines. Shares Supabase client and type definitions.

Pros

  • Clean, purpose-built for web
  • Full control over look/feel
  • No RN Web quirks
  • Easy to deploy to Cloudflare Pages

Cons

  • Some logic duplication
  • Separate deploy pipeline
C

Static shareable card only

Server-rendered page at deepr.fm/play — today's puzzle displayed as a beautiful card (start, end, category, blurred intermediates). Big "Download on App Store" CTA. No interactive game.

Pros

  • Fastest to build
  • Perfect OG/Twitter card

Cons

  • No "hook" — can't feel the mechanic
  • Less viral (no score to share)
